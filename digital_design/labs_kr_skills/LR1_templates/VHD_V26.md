### Разъяснение Задания 3: Блок `comb_led_unit`

В этом задании нужно взять два числа `I` и `J` из таблицы варианта, а затем разработать модуль, который, используя вход `sw_i` (8 бит) и число `I`, выполнит булеву операцию так, чтобы на выходе `led_o` (16 бит) получить число `J`.

**1. Выбор чисел I, J и представление их в двоичной форме:**

Для варианта 26, согласно Таблице 1:

- `I = 5C` (шестнадцатеричное)
- `J = 47` (шестнадцатеричное)

Переведем их в двоичный код:

- `I = 5C` (hex) = `0101_1100` (bin). Это 8-битное число.
- `J = 47` (hex) = `0100_0111` (bin). Это 8-битное число.

**2. Описание блока `comb_led_unit` в VHDL (`comb_led_unit.vhd`):**

Файл `comb_led_unit.vhd` реализует логику для этого задания.

- **Библиотеки:**

  ```vhdl
  library IEEE;
  use IEEE.STD_LOGIC_1164.ALL;
  ```

  Стандартная библиотека IEEE и пакет `STD_LOGIC_1164.ALL` для типов `std_logic` и `std_logic_vector`.

- **Сущность (Entity) `COMB_LED_UNIT`:**

  ```vhdl
  entity COMB_LED_UNIT is
      generic (
          INITVALUE : std_logic_vector := X"5C"; -- Это число I
          N : integer := 16
      );
      port (
          sw_in   : in std_logic_vector(INITVALUE'length - 1 downto 0); -- sw_in[7:0]
          led_out : out std_logic_vector(N - 1 downto 0)              -- led_o[15:0]
      );
  end COMB_LED_UNIT;
  ```

  - `generic (INITVALUE : std_logic_vector := X"5C"; N : integer := 16);`
    - `INITVALUE` теперь используется для хранения числа `I` (`X"5C"`). Это 8-битное значение.
    - `N` остается 16, так как `led_out` должен использовать все 16 светодиодов.
  - `port (` объявляет порты:
    - `sw_in : in std_logic_vector(INITVALUE'length - 1 downto 0);` - входной порт `sw_in`. Его размерность определяется длиной `INITVALUE`, то есть 8 бит (`sw_in[7:0]`).
    - `led_out : out std_logic_vector(N - 1 downto 0)` - выходной порт `led_out` имеет разрядность 16 бит (`led_o[15:0]`).

- **Архитектура (Architecture) `behavioral`:**

  ```vhdl
  architecture behavioral of COMB_LED_UNIT is
  constant INITVALUE_NORM  : std_logic_vector(INITVALUE'length - 1 downto 0) := INITVALUE;
  signal   int_value       : std_logic_vector(INITVALUE'length - 1 downto 0);
  begin
      int_value <= INITVALUE_NORM xor sw_in;
      led_out <= (N - 1 downto INITVALUE'length => '0') & int_value;
  end behavioral;
  ```

  - `constant INITVALUE_NORM : std_logic_vector(INITVALUE'length - 1 downto 0) := INITVALUE;` - локальная константа для числа `I`.
  - `signal int_value : std_logic_vector(INITVALUE'length - 1 downto 0);` - внутренний 8-битный сигнал, который будет хранить результат булевой операции.
  - `int_value <= INITVALUE_NORM xor sw_in;` - **ключевая булева операция**. Здесь выполняется побитовое исключающее ИЛИ (XOR) между числом `I` (`INITVALUE_NORM`) и входным портом `sw_in`. Результат этой операции записывается в `int_value`.
  - `led_out <= (N - 1 downto INITVALUE'length => '0') & int_value;` - это присваивание выходному порту `led_out`.
    - `INITVALUE'length` равно 8.
    - `N` равно 16.
    - `N - 1 downto INITVALUE'length` означает биты с 15 по 8.
    - `(N - 1 downto INITVALUE'length => '0')` создает 8-битный вектор из нулей (`"00000000"`). Это верхние 8 бит `led_out` (с `led_out(15)` по `led_out(8)`).
    - `& int_value` - оператор конкатенации (сцепления). Он объединяет вектор из нулей с 8-битным значением `int_value`.
    - Таким образом, `led_out` будет состоять из 8 нулей в старших разрядах и 8 бит `int_value` в младших разрядах. То есть, `led_out[15:8]` будут всегда '0', а `led_out[7:0]` будут показывать результат `I XOR sw_in`.

**3. Объяснение выбора преобразующей булевой функции:**

Задание требует, чтобы операция `(sw_i[7:0] <операция> I)` породила двоичный вектор, равный `J`.
То есть, мы ищем такую операцию `OP`, что `sw_i OP I = J`.
Известны `I` и `J`. Нам нужно найти `sw_i` (произвольно).
Если выбрана операция XOR: `sw_i XOR I = J`.
В алгебре логики известно, что `A XOR B = C` эквивалентно `A = C XOR B` и `B = C XOR A`.
Следовательно, `sw_i = J XOR I`.

Рассчитаем `J XOR I`:
`I = 0101_1100` (5C)
`J = 0100_0111` (47)
`J XOR I = (0100_0111) XOR (0101_1100)`
`  0100 0111`
`XOR 0101 1100`
`-----------`
`  0001 1011` (это 1B в шестнадцатеричном виде)

Таким образом, если мы подадим на `sw_in` значение `0001_1011` (1B), то `int_value` будет равно `J`:
`int_value = I XOR (J XOR I) = J`.
Следовательно, выбранная операция XOR позволяет достичь цели задания: при определенном (`произвольном`, но конкретно рассчитанном для отображения `J`) значении `sw_in`, на младших 8 светодиодах будет отображаться `J`. Старшие 8 светодиодов будут всегда выключены (по условию `led_out <= (N - 1 downto INITVALUE'length => '0') & int_value;`).

**4. Шаги в САПР Vivado и объяснение результатов:**

- **Подготовка описания физических ограничений (XDC-файл):**
  Необходимо сопоставить `sw_in[7:0]` с 8 физическими переключателями на Basys 3 (например, `SW0` - `SW7`).
  А также `led_out[15:0]` со всеми 16 физическими светодиодами (`LD0` - `LD15`).
  Пример строк в XDC:

  ```xdc
  set_property PACKAGE_PIN V17 [get_ports {sw_in[0]}]
  # ... для sw_in[1] по sw_in[7]

  set_property PACKAGE_PIN U16 [get_ports {led_out[0]}]
  # ... для led_out[1] по led_out[15]
  ```

- **Синтез:**
  Синтезатор Vivado преобразует оператор `int_value <= INITVALUE_NORM xor sw_in;` в 8 отдельных XOR-вентилей, каждый из которых будет выполнять операцию над соответствующими битами `INITVALUE_NORM` и `sw_in`.

**5. Структурная схема разработанного аппаратного обеспечения:**

Модуль `COMB_LED_UNIT` будет выглядеть как:

```
        sw_in[7] ---+
                    |
        sw_in[6] ---+
                    |
        sw_in[5] ---+
                    |
        sw_in[4] ---+
                    |
        sw_in[3] ---+          +-----+          +----- led_out[7]
                    |--------->| XOR |--------->|
        sw_in[2] ---+          |     |          +----- led_out[6]
                    |--------->| XOR |--------->|
```

Каждый XOR-вентиль берет один бит из `sw_in` и соответствующий бит из константы `I`, а затем выдает результат на соответствующий бит `led_out[7:0]`. Старшие 8 бит `led_out` (с `led_out[8]` по `led_out[15]`) просто подключаются к земле.

**2. Описание блока `function_unit` в VHDL (`function_unit.vhd`):**

Файл `function_unit.vhd` реализует эту логику. Он использует структурный метод, то есть описывает схему, инстанцируя (создавая экземпляры) более простых компонентов (NOT_1, AND_N, OR_N).

- **Библиотеки:**

  ```vhdl
  library IEEE;
  use IEEE.STD_LOGIC_1164.ALL;
  ```

  Стандартные библиотеки.

- **Сущность (Entity) `FUNCTION_UNIT`:**

  ```vhdl
  entity FUNCTION_UNIT is
      port (
          x_in  : in std_logic_vector(3 downto 0); -- Входные переменные F
          q_out : out std_logic                     -- Выход функции F
      );
  end FUNCTION_UNIT;
  ```

  - `x_in` - 4-битный входной вектор для `x3, x2, x1, x0`.
  - `q_out` - однобитный выход, результат функции `F`.

- **Архитектура (Architecture) `Structural`:**

  ```vhdl
  architecture Structural of FUNCTION_UNIT is
  component NOT_1 is
      port (
          a : in std_logic;
          q : out std_logic
      );
  end component;

  component AND_N is
      generic (
          N : integer := 4
      );
      port (
          x_in  : in std_logic_vector (N - 1 downto 0);
          q_out : out std_logic
      );
  end component;

  component OR_N is
      generic (
          N : integer := 4
      );
      port (
          x_in  : in std_logic_vector (N - 1 downto 0);
          q_out : out std_logic
      );
  end component;

  signal not_values : std_logic_vector(3 downto 0);
  signal int_value  : std_logic_vector(3 downto 0);

  begin
      NOT_N: for i in 3 downto 0 generate
          NOT_1_i: NOT_1
              port map (
                  a => x_in(i),
                  q => not_values(i)
              );
      end generate;

      TERM1: AND_N
          generic map (
              N => 3
          )
          port map (
              x_in(2) => not_values(3),
              x_in(1) => x_in(1),
              x_in(0) => x_in(0),
              q_out => int_value(3)
          );

      TERM2: AND_N
          generic map (
              N => 3
          )
          port map (
              x_in(2) => not_values(3),
              x_in(1) => not_values(1),
              x_in(0) => not_values(0),
              q_out => int_value(2)
          );

      TERM3: AND_N
          generic map (
              N => 2
          )
          port map (
              x_in(1) => not_values(3),
              x_in(0) => x_in(2),
              q_out => int_value(1)
          );

      TERM4: AND_N
          generic map (
              N => 2
          )
          port map (
              x_in(1) => x_in(2),
              x_in(0) => x_in(1),
              q_out => int_value(0)
          );

      FINALE_OR: OR_N
          port map (
              x_in(3) => int_value(3),
              x_in(2) => int_value(2),
              x_in(1) => int_value(1),
              x_in(0) => int_value(0),
              q_out => q_out
          );

  end Structural;
  ```

  - **Объявление компонентов:** В начале архитектуры объявляются компоненты `NOT_1`, `AND_N`, `OR_N`. Это как "импорт" других модулей VHDL, которые определены в отдельных файлах.
    - `NOT_1`: однобитный инвертор.
    - `AND_N`: N-входовой AND-вентиль.
    - `OR_N`: N-входовой OR-вентиль.
    - **Как это работает (переиспользование модулей):** Когда вы объявляете `component` в VHDL, вы говорите компилятору Vivado: "У меня есть внешний модуль с таким интерфейсом (портами и дженериками). Я буду использовать его здесь". Сам код этого компонента (например, `not_1.vhd`, `and_n.vhd`, `or_n.vhd`) должен быть добавлен в проект Vivado, чтобы синтезатор мог его найти и включить в общую схему. Vivado автоматически объединяет все модули, когда вы запускаете синтез.
  - **Сигналы:** `not_values` хранит инвертированные значения входных битов. `int_value` хранит промежуточные результаты (термы произведения).
  - **`NOT_N` (генерация инверторов):**
    `for i in 3 downto 0 generate ... end generate;` - это оператор VHDL для генерации повторяющихся экземпляров. Здесь создается 4 инвертора (экземпляра `NOT_1`), каждый из которых инвертирует соответствующий бит `x_in` и сохраняет результат в `not_values`.
    - `NOT_1_i: NOT_1 port map (a => x_in(i), q => not_values(i));` - это инстанцирование компонента `NOT_1`. `NOT_1_i` - это имя экземпляра. `port map` связывает порты компонента `NOT_1` с сигналами текущего модуля `FUNCTION_UNIT`.
  - **`TERM1`, `TERM2`, `TERM3`, `TERM4` (формирование термов произведения):**
    Здесь создаются экземпляры компонента `AND_N` для реализации термов произведения из минимизированной булевой функции.
    - `TERM1: AND_N generic map (N => 3) port map (... q_out => int_value(3));` - это экземпляр 3-входового AND-вентиля. Его входы подключаются к `not_values(3)` (т.е. `not x_in(3)`), `x_in(1)` и `x_in(0)`. Выход сохраняется в `int_value(3)`.
    - **Важно:** Каждое `port map` здесь соответствует одному терму произведения. Давайте сравним с нашей минимизированной функцией.
      - Первая строка: `TERM1: ... x_in(2) => not_values(3), x_in(1) => x_in(1), x_in(0) => x_in(0)`. Выглядит как `(not x_in(3) AND x_in(1) AND x_in(0))`.
      - Вторая строка: `TERM2: ... x_in(2) => not_values(3), x_in(1) => not_values(1), x_in(0) => not_values(0)`. Выглядит как `(not x_in(3) AND not x_in(1) AND not x_in(0))`.
      - Третья строка: `TERM3: ... x_in(1) => not_values(3), x_in(0) => x_in(2)`. Выглядит как `(not x_in(3) AND x_in(2))`.
      - Четвертая строка: `TERM4: ... x_in(1) => x_in(2), x_in(0) => x_in(1)`. Выглядит как `(x_in(2) AND x_in(1))`.

    **Стоп!** Моя минимизированная функция и то, что реализовано в коде, **не совпадают**. Это критический момент.
    Код реализует следующую функцию:
    `F = (not x_in(3) AND x_in(1) AND x_in(0))`
    `OR (not x_in(3) AND not x_in(1) AND not x_in(0))`
    `OR (not x_in(3) AND x_in(2))`
    `OR (x_in(2) AND x_in(1))`

    Давайте еще раз посмотрим на термы, которые реализованы в `function_unit.vhd`:
    - `TERM1 (N=3)`: `not x_in(3) AND x_in(1) AND x_in(0)`
    - `TERM2 (N=3)`: `not x_in(3) AND not x_in(1) AND not x_in(0)`
    - `TERM3 (N=2)`: `not x_in(3) AND x_in(2)`
    - `TERM4 (N=2)`: `x_in(2) AND x_in(1)`

    Давайте проверим, какая это функция, используя карту Карно:
    Термы:
    1.  `not x_in(3) and x_in(1) and x_in(0)` -> 0\_ \_11 : 0011, 0111
    2.  `not x_in(3) and not x_in(1) and not x_in(0)` -> 0\_ \_00 : 0000, 0100
    3.  `not x_in(3) and x_in(2)` -> 01\_ \_ : 0100, 0101, 0110, 0111
    4.  `x_in(2) and x_in(1)` -> _11_ : 0110, 0111, 1110, 1111

    Давайте построим карту Карно для **реализованной** функции:

    | x3x2 \ x1x0 | 00  | 01  | 10  | 11  |
    | :---------- | :-- | :-- | :-- | :-- | --------------------------------- |

### Разъяснение Задания 5: Блок `alu_unit` (Арифметико-логическое устройство)

Это задание направлено на создание простого арифметико-логического устройства (АЛУ), которое может выполнять различные операции над двумя входными операндами, выбираемыми с помощью переключателей.

**1. Выбор операций (A, B, C, D) для варианта 26:**

Для варианта 26, согласно Таблице 1, операции:

- `A = xor` (побитовое исключающее ИЛИ)
- `B = <<<3` (циклический сдвиг влево на 3 позиции)
- `C = nand` (побитовое НЕ-И)
- `D = -` (вычитание)

Операнды для этих операций: `sw[15:10]` и `sw[9:4]`. Результат `led_o[5:0]` будет 6-битным.

**2. Описание блока `alu_unit` в VHDL (`alu_unit.vhd`):**

Файл `alu_unit.vhd` реализует логику этого задания.

- **Библиотеки:**

  ```vhdl
  library IEEE;
  use IEEE.STD_LOGIC_1164.ALL;
  use IEEE.NUMERIC_STD.ALL;
  ```

  - `IEEE.STD_LOGIC_1164.ALL` для стандартных логических типов.
  - `IEEE.NUMERIC_STD.ALL` очень важен здесь, так как он предоставляет функции для работы с векторами как с числами (например, `to_integer`, `unsigned`, `to_unsigned`), что необходимо для арифметических (вычитание) и сдвиговых операций.

- **Сущность (Entity) `ALU_UNIT`:**

  ```vhdl
  entity ALU_UNIT is
      generic (
          N : integer := 16
      );
      port (
          sw_in   : in std_logic_vector(N - 1 downto 0);
          led_out : out std_logic_vector(N - 1 downto 0)
      );
  end ALU_UNIT;
  ```

  - `generic (N : integer := 16);` - разрядность портов по умолчанию 16 бит. Это удобно, так как Basys 3 имеет 16 переключателей и 16 светодиодов.
  - `port (` объявляет порты:
    - `sw_in : in std_logic_vector(N - 1 downto 0);` - входной порт, к которому будут подключены 16 переключателей.
    - `led_out : out std_logic_vector(N - 1 downto 0)` - выходной порт, к которому будут подключены 16 светодиодов.

- **Архитектура (Architecture) `behavioral`:**

  ```vhdl
  architecture behavioral of ALU_UNIT is
  signal int_value : std_logic_vector(5 downto 0); -- Результат 6-битных операций
  signal mask      : std_logic_vector(5 downto 0); -- Необычная маска для операции сдвига

  begin
      --mask <= "111111" srl (6 - to_integer(unsigned(sw_in(9 downto 4)))); -- Закомментированный вариант
      mask <= (2 downto 0 => sw_in(10), others => '0'); -- Реализованный вариант

      with sw_in(3 downto 0) select -- Выбор операции по sw_in[3:0]
          int_value <= sw_in(15 downto 10) xor sw_in(9 downto 4)                               when "0001", -- A: XOR
                       sw_in(15 downto 10) sll 3 or mask                                       when "0010", -- B: <<<3 (циклический сдвиг влево на 3)
                       sw_in(15 downto 10) nand sw_in(9 downto 4)                              when "0100", -- C: NAND
                       std_logic_vector(to_unsigned(to_integer(unsigned(sw_in(15 downto 10)))
                       - to_integer(unsigned(sw_in(9 downto 4))), 6))                          when "1000", -- D: - (вычитание)
                       "000000"                                                                when others; -- По умолчанию нули

      led_out <= (N - 1 downto 6 => '0') & int_value; -- Вывод результата на младшие 6 светодиодов
  end behavioral;
  ```

  - **Внутренние сигналы:**
    - `int_value : std_logic_vector(5 downto 0);` - это 6-битный сигнал, который будет хранить результат выбранной арифметической/логической операции. Это соответствует требованию `led_o[5:0]`.
    - `mask : std_logic_vector(5 downto 0);` - это сигнал, который используется в операции сдвига.
      - Закомментированная строка `mask <= "111111" srl (6 - to_integer(unsigned(sw_in(9 downto 4))));` предполагает, что маска генерируется на основе `sw_in(9 downto 4)`, что было бы более логично для гибкого сдвига.
      - **Реализованная строка:** `mask <= (2 downto 0 => sw_in(10), others => '0');`
        - Эта строка создает маску, где младшие 3 бита (`mask[2:0]`) принимают значение `sw_in(10)`, а остальные биты (`mask[5:3]`) равны '0'. Это довольно необычная и, возможно, ошибочная реализация для "циклического сдвига", так как она не зависит от величины сдвига (которая по заданию фиксирована на 3) и создает фиксированную маску, которая затем применяется через `OR`. **Этот `mask` не имеет отношения к циклическому сдвигу `<<<3`**. Скорее всего, это остаток от какой-то другой логики или ошибочная попытка реализовать специфическое поведение. Мы будем считать, что для `<<<3` маска не должна влиять на циклический сдвиг, а должна использоваться для других целей (или быть удаленной). Однако, поскольку она есть, нужно отметить ее влияние. Она будет устанавливать младшие 3 бита результата сдвига в '1', если `sw_in(10)` равен '1'.

  - **Выбор операции (`with sw_in(3 downto 0) select`):**
    Это оператор мультиплексирования. Он выбирает, какое выражение будет присвоено `int_value`, основываясь на значении 4-битного управляющего входа `sw_in(3 downto 0)`.
    - `sw_in[0]` управляет операцией A (XOR).
    - `sw_in[1]` управляет операцией B (сдвиг).
    - `sw_in[2]` управляет операцией C (NAND).
    - `sw_in[3]` управляет операцией D (вычитание).
    - Заметьте, что выбор операции осуществляется по битам `sw_in` с 0 по 3. Если, например, `sw_in(3 downto 0)` равно `"0001"`, выполняется XOR. Если `"0010"`, выполняется сдвиг. И так далее. Если несколько битов выбора активны одновременно, приоритет определяется порядком в `when` условиях.
    - `when others` - если ни одно из условий не совпадает, `int_value` становится `"000000"`.

  - **Реализация операций:**
    1.  **`A: XOR`** (при `sw_in(3 downto 0) = "0001"`):
        `sw_in(15 downto 10) xor sw_in(9 downto 4)`
        Выполняет побитовое XOR между двумя 6-битными операндами:
        - Операнд 1: `sw_in[15:10]` (старшие 6 бит от 16-битного входа)
        - Операнд 2: `sw_in[9:4]` (средние 6 бит от 16-битного входа)
          Результат - 6-битный вектор.

    2.  **`B: <<<3` (левый циклический сдвиг на 3)** (при `sw_in(3 downto 0) = "0010"`):
        `sw_in(15 downto 10) sll 3 or mask`
        - `sw_in(15 downto 10)` - первый операнд (6 бит).
        - `sll 3` - оператор логического сдвига влево на 3 позиции (shift left logical). **Важно:** `sll` в VHDL не является циклическим сдвигом по умолчанию. Он заполняет освободившиеся младшие биты нулями. Для циклического сдвига обычно требуется более сложная конструкция (например, `(operand(operand'length-4 downto 0) & operand(operand'length-1 downto operand'length-3))`).
        - В данном случае, если `sw_in(15 downto 10)` было `abcde_f_`, то `sll 3` даст `de_f_000`.
        - `or mask` - к результату логического сдвига применяется побитовое ИЛИ с `mask`. Как было отмечено выше, `mask` здесь зависит от `sw_in(10)` и никак не реализует циклический сдвиг. Если `sw_in(10)` равно '1', то младшие 3 бита результата станут '1', независимо от сдвига.
        - **Вывод:** Реализация для `<<<3` является логическим сдвигом влево на 3 с последующим ИЛИ с маской, которая не относится к циклическому сдвигу. Задание требует _циклического_ сдвига. Это еще одна потенциальная неточность в предоставленном коде.

    3.  **`C: NAND`** (при `sw_in(3 downto 0) = "0100"`):
        `sw_in(15 downto 10) nand sw_in(9 downto 4)`
        Выполняет побитовое НЕ-И между теми же 6-битными операндами.

    4.  **`D: -` (вычитание)** (при `sw_in(3 downto 0) = "1000"`):
        `std_logic_vector(to_unsigned(to_integer(unsigned(sw_in(15 downto 10))) - to_integer(unsigned(sw_in(9 downto 4))), 6))`
        - Это более сложная конструкция, так как VHDL не имеет встроенного оператора вычитания для `std_logic_vector` напрямую.
        - `unsigned(sw_in(15 downto 10))` - первый 6-битный операнд преобразуется в тип `unsigned`, что позволяет интерпретировать его как беззнаковое число.
        - `to_integer(...)` - преобразует `unsigned` в целое число (integer).
        - Выполняется стандартное целочисленное вычитание `-`.
        - `to_unsigned(..., 6)` - результат вычитания (целое число) преобразуется обратно в `unsigned` с указанной длиной 6 бит.
        - `std_logic_vector(...)` - окончательно преобразует `unsigned` обратно в `std_logic_vector`.
        - **Важно:** Если результат вычитания отрицательный или выходит за пределы 6-битного беззнакового числа, произойдет усечение/переполнение.

  - **Выходной порт `led_out`:**
    `led_out <= (N - 1 downto 6 => '0') & int_value;`
    - Как и в задании 3, старшие биты `led_out` (с `N-1` до `6`, т.е. `led_out[15:6]`) будут установлены в '0'.
    - Младшие 6 бит `led_out` (с `led_out[5]` до `led_out[0]`) будут отображать 6-битный результат `int_value`.

**3. Шаги в САПР Vivado и объяснение результатов:**

- **Подготовка описания физических ограничений (XDC-файл):**
  В этом задании `sw_in[15:0]` будут сопоставлены со всеми 16 физическими переключателями на Basys 3.
  `led_out[15:0]` будут сопоставлены со всеми 16 физическими светодиодами.
  Это потребует определения пинов для всех 16 `sw_in` и 16 `led_out` в XDC-файле.

- **Синтез:**
  Vivado проанализирует оператор `with select`. Это приведет к созданию мультиплексора (MUX), который будет выбирать один из четырех блоков комбинационной логики, соответствующих каждой операции (XOR, SLL+OR_mask, NAND, Subtractor).
  - XOR, NAND: будут синтезированы в соответствующие побитовые логические вентили (XOR, NAND).
  - SLL (сдвиг): будет реализован как простая перестановка проводников (для сдвига на константу 3) с добавлением OR-вентилей для `mask`.
  - Subtractor: `NUMERIC_STD` позволяет синтезатору реализовать полноценный 6-битный вычитатель (на основе полных вычитателей и заимствований).
  - Мультиплексор: будет использовать `sw_in(3 downto 0)` как входы выбора, чтобы направить выход нужной операции на `int_value`.

- **Имплементация, Генерация битстрима:**

### Разъяснение Задания 6: Преобразователь кодов из G в L (функциональный метод)

Это означает, что концепция "Бегущая 1 (4 бит)" в задании, по всей видимости, является упрощенным описанием, а реальная реализация рассчитана на 9-битный вход, чтобы охватить цифры от 1 до 9. Мы будем исходить из того, что файл `code_translator_unit.vhd` является правильной реализацией для этого задания.
  - Входные паттерны "Бегущая 1" для цифр 1-9 (где `sw_in[0]` соответствует "1", `sw_in[1]` - "2" и т.д.):

- **L = Айкена:** Это 4-битный взвешенный BCD-код (2-4-2-1).

  - 8: `1110`

**2. Описание преобразователя кодов в VHDL (`code_translator_unit.vhd`):**

Файл `code_translator_unit.vhd` реализует этот преобразователь.

- **Библиотеки:**

  ```vhdl
  library IEEE;
  use IEEE.STD_LOGIC_1164.ALL;
  ```

  Стандартные библиотеки VHDL.

- **Сущность (Entity) `CODE_TRANSLATOR_UNIT`:**

  ```vhdl
  entity CODE_TRANSLATOR_UNIT is
      generic (
          N : integer := 16
      );
      port (
          sw_in   : in std_logic_vector(N - 1 downto 0);
          led_out : out std_logic_vector(N - 1 downto 0)
      );
  end CODE_TRANSLATOR_UNIT;
  ```

  - `generic (N : integer := 16);` - параметр разрядности портов, по умолчанию 16 бит.
  - `sw_in` (16 бит) - входной порт, к которому будут подключены переключатели.
  - `led_out` (16 бит) - выходной порт, к которому будут подключены светодиоды.

- **Архитектура (Architecture) `behavioral`:**

  ```vhdl
  architecture behavioral of CODE_TRANSLATOR_UNIT is
  signal int_value : std_logic_vector (3 downto 0);

  begin
      with sw_in(8 downto 0) select -- Выбор выходного кода L на основе 9-битного входа G
          int_value <= "0001" when "000000001", -- G (1-ый бит) -> L (1)
                       "0010" when "000000010", -- G (2-ой бит) -> L (2)
                       "0011" when "000000100", -- G (3-ий бит) -> L (3)
                       "0100" when "000001000", -- G (4-ый бит) -> L (4)
                       "1011" when "000010000", -- G (5-ый бит) -> L (5)
                       "1100" when "000100000", -- G (6-ой бит) -> L (6)
                       "1101" when "001000000", -- G (7-ой бит) -> L (7)
                       "1110" when "010000000", -- G (8-ой бит) -> L (8)
                       "1111" when "100000000", -- G (9-ый бит) -> L (9)
                       "0000" when others;     -- Если вход не соответствует, выводим L (0)

  led_out <= (N - 1 downto 4 => '0') & int_value; -- Вывод 4-битного результата на младшие светодиоды
  end behavioral;
  ```

  - `signal int_value : std_logic_vector (3 downto 0);` - внутренний 4-битный сигнал, который будет хранить преобразованный код Айкена.
  - `with sw_in(8 downto 0) select ... when others;` - это **функциональный метод** описания. Он напрямую описывает зависимость выхода от входа с помощью таблицы истинности (или условий). В данном случае, он:
    - Берет 9-битный срез входного вектора `sw_in(8 downto 0)`. Это и есть вход G.
    - Каждое `when` условие сопоставляет конкретный 9-битный шаблон "Бегущей 1" (например, `"000000001"`) с соответствующим 4-битным кодом Айкена (например, `"0001"`).
    - `when others` - это важный пункт, который определяет поведение преобразователя для всех остальных 9-битных входных комбинаций (например, если ни один бит не установлен, или установлено несколько битов). В данном случае, он выводит `0000` (код Айкена для 0). Это обеспечивает полное определение функции.
  - `led_out <= (N - 1 downto 4 => '0') & int_value;` - как и в предыдущих заданиях, 4-битный результат `int_value` выводится на младшие светодиоды (`led_out[3:0]`), а остальные (старшие) светодиоды (`led_out[15:4]`) будут выключены ('0').

**3. Шаги в САПР Vivado и объяснение результатов:**

- **Подготовка описания физических ограничений (XDC-файл):**
  - Необходимо сопоставить вход `sw_in[8:0]` с 9 физическими переключателями на плате Basys 3 (например, `SW0` - `SW8`).
  - Выход `led_out[3:0]` нужно сопоставить с 4 физическими светодиодами (например, `LD0` - `LD3`).
  - Остальные переключатели и светодиоды, не используемые в этом модуле, могут быть либо не сопоставлены (Vivado их оптимизирует), либо также сопоставлены для других целей.

- **Синтез:**
  Vivado преобразует оператор `with select` в комбинационную логику. Для каждого `when` условия будет создан декодер (набор AND-вентилей), который обнаруживает соответствующий входной шаблон. Затем выходы этих декодеров будут поданы на мультиплексор, который выберет правильный 4-битный код Айкена. Поскольку входные шаблоны являются "one-hot", декодеры будут относительно простыми.
  - Анализ схемы через `schematic tool` покажет набор AND-вентилей, которые определяют, какой из 9 входных шаблонов "Бегущей 1" активен. Выходы этих AND-вентилей затем будут управлять мультиплексорами, которые формируют 4-битный выходной код Айкена.

- **Имплементация, Генерация битстрима:**
  Стандартные шаги по размещению логических элементов на кристалле ПЛИС, их трассировке и генерации файла конфигурации (.bit).

- **Загрузка прошивки и эмпирические результаты:**
  После загрузки файла `.bit` в ПЛИС на плате Basys 3:
  - Вы активируете один из переключателей `SW0` - `SW8` (например, переводите `SW0` в '1', а остальные `SW1-SW8` в '0').
  - На светодиодах `LD0` - `LD3` вы увидите соответствующий 4-битный код Айкена.
    - Если `SW0='1'`, `LD0-LD3` покажет `0001` (Айкена для 1).
    - Если `SW1='1'`, `LD0-LD3` покажет `0010` (Айкена для 2).
    - ...
    - Если `SW8='1'`, `LD0-LD3` покажет `1111` (Айкена для 9).
  - Если вы активируете несколько переключателей `SW0` - `SW8` одновременно, или ни одного из них, или переключатель за пределами `SW0-SW8`, то `LD0-LD3` покажут `0000` (Айкена для 0), согласно `when others`.
  - Светодиоды `LD4` - `LD15` будут всегда выключены.

**4. Структурная схема разработанного аппаратного обеспечения:**

На высоком уровне это выглядит как "черный ящик" преобразователя кодов:

```

sw_in[1] -------------------| (например, AND-gate |
...                         | для "00...001")    |
sw_in[8] -------------------|                    |
                            +---------+----------+
                                      |
                                      V (несколько выходов, по одному на каждый шаблон G)
```

**Критический путь:**
Критический путь - это самый длинный путь распространения сигнала от входа до выхода, который определяет максимальную частоту работы схемы. В данном случае, для комбинационной схемы, это будет путь от любого бита `sw_in[8:0]` через логику декодирования и затем через мультиплексор до любого бита `led_out[3:0]`. Например, от `sw_in[8]` через AND-вентиль, затем через мультиплексор до `led_out[3]`. Длина этого пути (количество вентилей/задержек) будет зависеть от сложности логики, сгенерированной синтезатором для `with select` оператора. Визуально на схеме (в Vivado Schematic Tool) он будет выглядеть как последовательность соединенных логических элементов, идущих от входа к выходу.

### Разъяснение Задания 7: Преобразователь кодов из G в L (структурный метод, минимизация с `mux2` и инверторами)

Это задание по своей сути очень похоже на Задание 6 (преобразователь кодов из G в L), но с двумя ключевыми отличиями:

1.  **Минимизация с учетом избыточных наборов:** Требуется оптимизировать схему по количеству вентилей, используя "избыточные наборы" (don't care conditions) для упрощения логики.
2.  **Структурный метод описания:** Разрешено использовать только вентили `mux2` (2-входовой мультиплексор) и инверторы (`not_1`).

Файл `cccode_translator_unit.vhd` является реализацией для этого задания.

**1. Определение кодов G и L и "Избыточные наборы":**

Как и в Задании 6:

- **G = Бегущая 1 (4 бит)**: Интерпретируется как 9-битный "one-hot" код для цифр 1-9.
  - `sw_in[8:0]`: `000000001` (для 1), `000000010` (для 2), ..., `100000000` (для 9).
- **L = Айкена (4 бит)**:
  - 0: `0000`, 1: `0001`, 2: `0010`, 3: `0011`, 4: `0100`, 5: `1011`, 6: `1100`, 7: `1101`, 8: `1110`, 9: `1111`.

**Избыточные наборы (Don't care conditions):**
Входной код "Бегущая 1" подразумевает, что _только один_ бит `sw_in[8:0]` может быть активен ('1') в любой момент времени. Все остальные `2^9 - 9 = 503` комбинации входного `sw_in[8:0]` являются "избыточными" или "запрещенными" наборами. Это означает, что для этих комбинаций нам неважно, что будет на выходе, и мы можем использовать это для упрощения логической функции (на картах Карно это обозначается 'X' или 'd').

**2. Минимизация и построение функций:**

В этом задании мы должны вывести 4 логические функции (по одной для каждого выходного бита кода Айкена `L3, L2, L1, L0`) в зависимости от 9 входных переменных `S8, S7, ..., S0`.

Давайте посмотрим на логику, реализованную в `cccode_translator_unit.vhd`.
Она использует следующий подход:
а. **Декодирование "one-hot" входа:**
Создаются 9 сигналов `minterms(i)`, где `minterms(i)` будет '1' _только в том случае_, если бит `sw_in(i)` активен, а все остальные биты `sw_in[8:0]` неактивны.
Например, `minterms(0)` будет '1' только когда `sw_in[0] = '1'` и `sw_in[1..8] = "00000000"`.
Эта логика реализована в цикле `AND_N_ARRAY`, используя компонент `AND_N_MUX` (который, как мы выяснили, построен из `MUX_2` и `NOT_1`). Каждый `minterms(i)` по сути является 9-входовым И-вентилем, который проверяет один конкретный one-hot шаблон.

б. **Формирование выходных битов Айкена:**
Каждый бит выходного кода Айкена `L_k` формируется как логическое ИЛИ (OR) из тех `minterms(i)`, для которых `L_k` должен быть '1'.
_ `L3` (соответствует `out_value(3)`) активен для цифр 5, 6, 7, 8, 9. В коде это:
`out_value(3) <= minterms(8) OR minterms(7) OR minterms(6) OR minterms(5) OR minterms(4);`
(что эквивалентно: `minterms(4)` для 5, `minterms(5)` для 6, `minterms(6)` для 7, `minterms(7)` для 8, `minterms(8)` для 9)
_ `L2` (соответствует `out_value(2)`) активен для цифр 4, 6, 7, 8, 9. В коде это:
`out_value(2) <= minterms(8) OR minterms(7) OR minterms(6) OR minterms(5) OR minterms(3);`
(что эквивалентно: `minterms(3)` для 4, `minterms(5)` для 6, `minterms(6)` для 7, `minterms(7)` для 8, `minterms(8)` для 9) \* Аналогично для `L1` и `L0`.

**Как это связано с минимизацией и "избыточными наборами":**
Эта реализация фактически вычисляет каноническую сумму минтермов для каждого выходного бита, учитывая _только_ те входные комбинации, которые являются "легальными" one-hot кодами. Все "избыточные наборы" (т.е. когда вход `sw_in[8:0]` не является one-hot кодом) автоматически приводят к тому, что _все_ `minterms(i)` будут '0'. Следовательно, и все выходные биты `L3..L0` также будут '0' (код Айкена для '0'). Это **является формой минимизации**, так как мы не тратим логические вентили на обработку этих "избыточных" комбинаций для получения какого-либо специфического (но ненужного) значения. Вместо этого, они естественным образом отображаются в '0'.

- **Библиотеки:**

  Стандартный интерфейс с 16-битными входными/выходными портами.

- **Архитектура `structural`:**

  ```vhdl

  signal in_value     : std_logic_vector(8 downto 0);
  signal out_value    : std_logic_vector(3 downto 0);
  signal inv_in_value : std_logic_vector(8 downto 0);
  signal minterms     : std_logic_vector(8 downto 0);

  -- Промежуточные сигналы для OR-вентилей
  signal out_bit_3_minterms : std_logic_vector(4 downto 0);
  signal out_bit_2_minterms : std_logic_vector(4 downto 0);
  signal out_bit_1_minterms : std_logic_vector(4 downto 0);
  signal out_bit_0_minterms : std_logic_vector(4 downto 0);

  begin
      in_value <= sw_in(8 downto 0); -- Берем младшие 9 бит переключателей

      -- Генерация инвертированных входных сигналов
      NOT_N: for i in 8 downto 0 generate
          NOT_1_Ni: NOT_1
              port map (
                  a => in_value(i),
                  q => inv_in_value(i)
              );
      end generate;

      -- Генерация one-hot минтермов (декодирование входа G)
      AND_N_ARRAY: for i in 8 downto 0 generate
          signal temp : std_logic_vector(8 downto 0);
      begin
           -- temp - это вектор, где sw_in(i) - прямой, остальные - инвертированные
           temp <= inv_in_value(8 downto i+1) & in_value(i) & inv_in_value(i-1 downto 0);
          ARN_N_Ni: AND_N_MUX
              generic map (
                  N => 9
              )
              port map (
                  x_in  => temp,
                  q_out => minterms(i) -- minterms(i) активен, если только sw_in(i) = '1'
              );
      end generate;

      -- Назначение минтермов для каждого выходного бита L
      out_bit_3_minterms <= (4 => minterms(8), 3 => minterms(7), 2 => minterms(6), 1 => minterms(5), 0 => minterms(4));
      out_bit_2_minterms <= (4 => minterms(8), 3 => minterms(7), 2 => minterms(6), 1 => minterms(5), 0 => minterms(3));
      out_bit_1_minterms <= (4 => minterms(8), 3 => minterms(7), 2 => minterms(2), 1 => minterms(1), 0 => minterms(4));
      out_bit_0_minterms <= (4 => minterms(8), 3 => minterms(6), 2 => minterms(4), 1 => minterms(2), 0 => minterms(0));

      -- Объединение минтермов через OR-вентили для получения выходных битов L
      OR_N_N3: OR_N_MUX
          generic map (N => 5)
          port map (x_in  => out_bit_3_minterms, q_out => out_value(3));
      OR_N_N2: OR_N_MUX
          generic map (N => 5)
          port map (x_in  => out_bit_2_minterms, q_out => out_value(2));
      OR_N_N1: OR_N_MUX
          generic map (N => 5)
          port map (x_in  => out_bit_1_minterms, q_out => out_value(1));
      OR_N_N0: OR_N_MUX
          generic map (N => 5)
          port map (x_in  => out_bit_0_minterms, q_out => out_value(0));

      -- Вывод 4-битного результата на младшие светодиоды
      led_out <= (N - 1 downto 4 => '0') & out_value;

  end structural;
  ```

  - **Использование компонентов `MUX_2` и `NOT_1`:**
    Как мы выяснили в предыдущих обсуждениях, компоненты `AND_N_MUX` и `OR_N_MUX` сами по себе построены из `MUX_2` и `NOT_1` (например, `AND_2_MUX` является `MUX_2` с одним входом на '0', а `OR_N_MUX` реализует `NOT(AND(NOT inputs))` по законам Де Моргана, используя `NOT_1` и `AND_N_MUX`). Таким образом, требование использования только `mux2` и инверторов технически соблюдено, так как все более сложные вентили синтезируются из этих примитивов.
  - **Структурный подход:** Код явно инстанциирует (`port map`) компоненты `NOT_1`, `AND_N_MUX` и `OR_N_MUX`, соединяя их сигналами. Это является чистым структурным описанием схемы.

**4. Шаги в САПР Vivado и объяснение результатов:**

- **Подготовка описания физических ограничений (XDC-файл):**
  - Так же, как в Задании 6, `sw_in[8:0]` сопоставляется с 9 физическими переключателями (например, `SW0` - `SW8`).
  - `led_out[3:0]` сопоставляется с 4 физическими светодиодами (например, `LD0` - `LD3`).

- **Синтез:**
  Vivado преобразует все экземпляры компонентов (`NOT_1`, `MUX_2`, `AND_N_MUX`, `OR_N_MUX`) в базовые логические элементы (LUTs, FLOPs), доступные на ПЛИС Artix-7.
  - `NOT_1` превратится в инвертор.
  - `MUX_2` (2-входовой мультиплексор) будет реализован с помощью одного 2-входового LUT, что является очень эффективным.
  - `AND_N_MUX` (который является цепочкой `AND_2_MUX` через `MUX_2`) и `OR_N_MUX` (также цепочка `AND_N_MUX` через `MUX_2` и `NOT_1`) будут синтезированы в последовательность LUTs.
  - Анализ схемы через `schematic tool` покажет детальную структуру из соединенных `MUX_2` и `NOT_1` (или LUTs), формирующих логику декодирования и кодирования. Вы увидите 9 9-входовых AND-блоков (реализованных как цепочки `MUX_2`/LUTs) и 4 5-входовых OR-блока (также цепочки `MUX_2`/LUTs).

- **Имплементация, Генерация битстрима:**
  Стандартные шаги по размещению и трассировке логических элементов на кристалле ПЛИС, создание файла конфигурации.

- **Загрузка прошивки и эмпирические результаты:**
  После загрузки файла `.bit` в ПЛИС:
  - Поведение будет идентично Заданию 6.
  - Когда вы активируете один переключатель `SWi` (например, `SW0` для цифры 1), остальные `SWj` (`j` не равно `i`) должны быть выключены.
  - Светодиоды `LD0` - `LD3` покажут соответствующий 4-битный код Айкена. Например, `SW0='1'` -> `LD0-LD3 = 0001`. `SW8='1'` -> `LD0-LD3 = 1111`.
  - Если входной `sw_in[8:0]` не является one-hot кодом (например, несколько переключателей активны или ни один не активен), то светодиоды `LD0` - `LD3` покажут `0000` (код Айкена для 0).
  - Светодиоды `LD4` - `LD15` будут всегда выключены.

**5. Структурная схема разработанного аппаратного обеспечения и критический путь:**

- **Структурная схема:**
  На верхнем уровне схема выглядит так же, как в Задании 6. Но детализация внутри блока `CCCODE_TRANSLATOR_UNIT` очень важна для Задания 7.

  ```
                    sw_in[8] ---+---> NOT_1 --- inv_in_value[8] ---+
                    sw_in[7] ---+---> NOT_1 --- inv_in_value[7] ---|
                    ...                                             |
                    sw_in[0] ---+---> NOT_1 --- inv_in_value[0] ---+
                                                                   |
                                                                   V
                              +------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
                              |                                                                                                                                                                                            |
                              |  +--------------------+   +--------------------+   +--------------------+                                                                                                                    |
                              |  | AND_N_MUX (N=9)    |   | AND_N_MUX (N=9)    |   | AND_N_MUX (N=9)    |                                                                                                                    |
                              +->| (для sw_in[0]=1)   |-->| (для sw_in[1]=1)   |-->| (для sw_in[2]=1)   |--> ... (всего 9 таких блоков) ... -->| AND_N_MUX (N=9)    |                                             |
                                 | (minterms(0))      |   | (minterms(1))      |   | (minterms(2))      |                                      | (для sw_in[8]=1)   |----------------+
                                 +--------------------+   +--------------------+   +--------------------+                                      | (minterms(8))      |                |
                                                                                                                                                                                         |                |
                                                                                                                                                                                         V                V
                                                                                                                                               minterms[8] --+                minterms[8] --+
                                                                                                                                               minterms[7] --|                minterms[7] --|
                                                                                                                                               minterms[6] --|                minterms[6] --|
                                                                                                                                               minterms[5] --|   +----------+   minterms[5] --|   +----------+
                                                                                                                                               minterms[4] --+->| OR_N_MUX |   minterms[4] --+->| OR_N_MUX |
                                                                                                                                               minterms[3] ---  | (N=5)    |                 | (N=5)    |
                                                                                                                                               minterms[2] ---  +----------+                 | +----------+
                                                                                                                                               minterms[1] ---      |                        minterms[3] --|
                                                                                                                                               minterms[0] ---      |                        minterms[2] --|
                                                                                                                                                                    V                        minterms[1] --|
                                                                                                                                                                                            minterms[0] --|
                                                                                                                                                                                                              |
                                                                                                                                                          out_value[3] <--------------------+   out_value[2] <-------+
                                                                                                                                                          out_value[1] <--------------------+   out_value[0] <-------+

                                                                                                                                                                                                              |
                                                                                                                                                                                                              V
                                                                                                                                                                                                    led_out[3:0]
                                                                                                                                                                                                    led_out[15:4] (GND)
  ```

  _Каждый `AND_N_MUX` и `OR_N_MUX` сам состоит из цепочки `MUX_2` и `NOT_1`._

- **Критический путь:**
  Поскольку `AND_N_MUX` и `OR_N_MUX` реализованы как последовательные цепочки 2-входовых вентилей (а не как сбалансированные деревья), критический путь будет достаточно длинным.
  1.  Начинается от входа `sw_in(i)`.
  2.  Возможно, проходит через `NOT_1` (для создания `inv_in_value(i)`).
  3.  Затем проходит через 8 последовательных `AND_2_MUX` внутри `AND_N_MUX` (для `N=9`) для формирования `minterms(i)`.
  4.  Затем проходит через 4 последовательных `OR_2_MUX` (если `OR_N_MUX` реализован как цепочка `OR_2`) внутри `OR_N_MUX` (для `N=5`) для формирования соответствующего `out_value(k)`. (Вспомним, что `OR_N_MUX` использует `AND_N_MUX` и `NOT_1` - так что это еще больше увеличивает задержку.)
      **Примерный критический путь (для `L0` через `minterms(8)`):**
      `sw_in[8]` -> `NOT_1` (если `sw_in[j]` для `j!=8` инвертируется) -> `AND_2_MUX_1` -> `AND_2_MUX_2` -> ... -> `AND_2_MUX_8` (для `minterms(8)`) -> `NOT_1` (из `OR_N_MUX`) -> `AND_2_MUX_1` -> ... -> `AND_2_MUX_4` (из `OR_N_MUX`) -> `NOT_1` (из `OR_N_MUX`) -> `led_out[0]`.
      Это очень длинная последовательность из десятков вентилей `MUX_2` и `NOT_1`. Именно такая цепочечная структура и будет отображена на схеме, выделяя самый длинный путь.

Это задание очень хорошо показывает, как даже простая функция может быть реализована структурно с использованием базовых элементов, и как выбор реализации (цепочка против дерева) влияет на критический путь и производительность.
